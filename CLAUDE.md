# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Bubble Splash is a Flutter game with a Flame-powered core and a full meta-game around it: a core loop (pop → score → XP → progression), Home/Profile/Leaderboard/Shop/Results screens, and retention hooks (lives-as-continues, watch-ad-for-life, the "Free Life" cooldown reward, achievements, leaderboard). Built with `flame` 1.37 + `flutter_riverpod` 3. Enabled platforms: Android, iOS — both scaffolded (`android/`, `ios/`). iOS was added via `flutter create --platforms=ios .`; iOS bundle id is `com.bubblesplash.game` (Android applicationId: `com.bubblesplash.game`). iOS **compile/run requires macOS + Xcode**; this repo is developed on Windows, so iOS can be scaffolded/committed here but only built on a Mac. Dart code is platform-agnostic (no `Platform.`/`dart:io`/`MethodChannel` branches).

**Economy model (important — easy to get wrong, was deliberately changed):** Play is **always free**; lives never gate starting a round. A life is an in-round **continue**: when round HP depletes the player spends a banked life (or watches an ad) to revive. Lives are earned passively (1 per 30 min, timestamp-based) and via the **Free Life** ad claim (`FreeLifeController`, 30-min cooldown), banked up to `LivesState.maxLives` (10). **Coins are NOT earned by playing** — they're a *purchasable* currency (`PurchaseService`, fake impl now / RevenueCat later) spent only on Shop skins. A round grants XP/progression only.

## Commands

```bash
flutter pub get                 # fetch dependencies
flutter run -d emulator-5554    # run on Android emulator
flutter run --profile -d emulator-5554   # profile build — the ONLY valid way to judge fps; watch `app_time_stats` (~16ms budget). Debug overstates jank.
flutter analyze                 # static analysis (lints from analysis_options.yaml)
flutter test                    # run all tests
flutter test --plain-name "popping a bubble increases the score"   # run a single test by name
dart run tool/gen_audio.dart    # regenerate assets/audio/*.wav sound effects
adb -s emulator-5554 shell pm clear com.bubblesplash.game   # wipe saved state (profile/lives/free-life) to verify a fresh install
adb -s emulator-5554 exec-out screencap -p > shot.png        # headless visual check; `adb shell input tap X Y` drives the UI (screen 1440x3120)
```

After changing pure Dart/Flutter code, hot reload (`r` in the `flutter run` session) is enough. Adding a **dependency or asset** requires a full restart (`flutter run` again), not hot reload. All persistent state lives in `SharedPreferences` (keys `profile`/`lives`/`free_life`); `pm clear` (above) is the way to reset to a brand-new-player state for manual verification.

## Architecture

Layered, with strict dependency direction `presentation → application → domain ← data`, plus an isolated `game/`. Each layer maps to a folder under `lib/`:

- **`domain/`** — pure Dart, no Flutter/plugin imports. Models (`PlayerProfile`, `LivesState`, `GameResult`/`RewardSummary`, `LeaderboardEntry`, `FreeLifeState`, `BubbleSkin`, `Achievement`), repository/service **interfaces** (incl. `PurchaseService` + `CoinPack` catalog), and catalogs. Achievement unlock conditions and the level math live here as pure functions, so they're trivially unit-testable. Colors are stored as ARGB `int` to keep Flutter out.
- **`data/`** — implements the domain interfaces: `local/` prefs-backed repos (sync), `fake/` simulated-remote leaderboard (seeded, async with latency), `services/` fake rewarded-ad + fake purchase (IAP) + no-op notification scheduler.
- **`application/`** — Riverpod controllers own all meta-state: `ProfileController`, `LivesController`, `FreeLifeController`, `leaderboardProvider`, `GameSessionController`. **`providers.dart` is the single swap point** — the only place concrete `data/` implementations are named. Swapping the mocks for Firebase/AdMob/RevenueCat means editing only that file.
- **`presentation/`** — `ConsumerWidget` screens (`home`, `game`, `profile`, `leaderboard`, `shop`) and widgets. No widget reads a repository directly; everything goes through controllers. **Home (`home_screen.dart`) was rebuilt to an external design handoff ("Candy Cosmos"):** cosmic-violet background + pink/orange nebula glows, four glossy floating bubbles (each bobbing on its own duration so the cluster never visibly repeats), a glowing `google_fonts` **Baloo 2** wordmark, glass stat/nav pills, and a warm orange 3D PLAY. Every px value is from the handoff's **322px-wide reference frame** and multiplied by a `_s(context) = screenWidth / 322` scale factor — when editing Home, keep new dimensions in that same `* s` space, don't hardcode logical px. Home is now the redesigned reference; the other screens still use the older glass widgets (`glass.dart`, `status_badges.dart`), so the UI is mid-migration — don't assume one shared theme yet.
- **`game/`** — the Flame engine, kept **Riverpod-free**. `GameScreen` injects `onGameOver(GameResult)`, `onContinueOffer()`, and the equipped skin's palette. The game reports outcomes up; the screen bridges to controllers and shows the continue sheet. In-round state (`score`, `hp`, `combo`, `soundOn`) stays as `ValueNotifier`s consumed by `GameHud`/`ResultsOverlay`.

Key flows:
- **Sync vs async repositories:** profile/lives/free-life repos are **synchronous** (prefs is warmed in `main()` and injected via `sharedPreferencesProvider.overrideWithValue`), so meta screens never flash a loading state and Riverpod `Notifier.build()` can read them directly. Only the leaderboard repo is async (it models the network) and is consumed via `FutureProvider` with `.when(...)`.
- **Leaderboard is two-metric:** boards rank by a `LeaderboardMetric` — `highScore` (peak single-round score) or `totalPops` (lifetime bubbles popped) — crossed with `LeaderboardScope` (local/global). `leaderboardProvider` is a `FutureProvider.autoDispose.family` keyed on the record `(scope, metric)`; the repo only supplies bots per scope, and the **controller** merges the player, sorts by `entry.valueFor(metric)`, and assigns ranks (sorting lives in the controller, not the repo). Fake bots seed `bubblesPopped` loosely correlated to `score` with heavy jitter so the two boards rank people differently. Add a new ranking metric by extending the enum + `valueFor`, not by adding a repo method.
- **Continue flow:** depleting round HP (or popping a bomb) calls `_offerContinue()` → `pauseEngine()` (guarded by `isMounted` so headless tests don't touch a missing game loop) + `onContinueOffer()`. The screen shows `continue_round_sheet.dart`; the player continues with a banked life or watches a rewarded ad (each ad banks +1 life, capped at 3 ads per prompt), then `game.continueRound()` restores HP, **clears all on-screen bubbles**, and grants a `headStartSeconds` (3s) breather — difficulty (score-derived spawn speed) is preserved, only the screen resets. The `headStart` `ValueNotifier` drives the full-screen "GET READY 3·2·1" overlay (`_HeadStartOverlay` in `game_screen.dart`). Declining calls `game.finishRound()` (emits the `GameResult`). `_awaitingDecision` freezes taps/spawns while the sheet is up.
- **Lives regen is timestamp-based** (`LivesState.lastRegenAtMs`): `LivesController` recomputes earned lives from elapsed time on load and on a 1-second timer, so regen is correct while the app is closed. `FreeLifeController` uses the same timestamp approach for its 30-min ad-claim cooldown (`FreeLifeState.lastClaimMs`). A separate `livesTickerProvider` drives per-second countdown UIs without churning controller state. `clockProvider` injects the clock so time logic is deterministically testable.
- **Bubble kinds:** `Bubble` carries a `BubbleKind` (normal/golden/bomb). Golden = bonus points; popping a bomb depletes the round (continue offered); letting a bomb escape is harmless. Combo multiplier rises with consecutive pops within `comboWindow`; each scoring pop spawns a floating `ScorePopup` ("xN") on the game root.
- **Avatars are Material icons, not emoji** (`player_avatar.dart`, `kAvatarIcons`/`avatarIconFor`). Old Androids often lack a color-emoji font and render blank tofu boxes, so the profile `avatarEmoji` field stores an icon *key* string. Don't reintroduce raw emoji glyphs in rendered UI for the same reason.
- **Player names are unique** (`ProfileController`): a stable 4-digit `#tag` derived from the profile id is appended (Discord-style), so duplicate base names never collide. Set on the fresh default and re-applied on every `rename`.

- **Coordinate space gotcha:** bubbles and pop effects are added directly to the game root (`game.add(...)`), **not** to `game.world`. They're positioned in screen coordinates using `size`. Adding a visual to `game.world` would put it in camera/world space and render in the wrong place — keep new gameplay visuals on the game root.

### Headless-test safety

Tests drive controllers and the Flame game directly without a widget tree, so anything needing platform plugins must be guarded:

- Controller/repository tests use a `ProviderContainer` with `sharedPreferencesProvider.overrideWithValue(...)` (`SharedPreferences.setMockInitialValues({})`) and `clockProvider.overrideWithValue(() => now)` for deterministic time. See `test/lives_controller_test.dart` for the pattern.
- The game's audio (`FlameAudio`) is guarded by `isMounted` + `try/catch` so it no-ops headlessly. **Preserve these guards** when adding sound, or `test/widget_test.dart` will crash on missing plugins.

### Audio assets

Sound effects are generated, not committed by hand: [tool/gen_audio.dart](tool/gen_audio.dart) synthesizes `assets/audio/pop.wav` and `game_over.wav` from pure math using only the Dart SDK. Edit the synthesis functions there and rerun `dart run tool/gen_audio.dart` to change the sounds. `assets/audio/` is registered in `pubspec.yaml`; `FlameAudio`'s default prefix is `assets/audio/`.

## Gotchas & lessons learned

Hard-won rules from building this — follow them when adding features to avoid repeating the same bugs:

- **Never mutate a Riverpod provider inside a widget life-cycle** (`initState`, `build`, `didChangeDependencies`, `dispose`). Doing so throws *"Tried to modify a provider while the widget tree was building."* This bit us when `GameScreen.initState` called `livesController.spendLife()` (which sets state). Fix pattern used in [game_screen.dart](lib/presentation/screens/game_screen.dart): defer the mutation with `WidgetsBinding.instance.addPostFrameCallback((_) { if (mounted) ... })`, or do it in a callback (e.g. a button's `onPressed`). **Reading** a provider in `build` (`ref.watch`/`ref.read`) is fine — only *mutation* is forbidden.
- **Don't make the sync repos async to "be safe."** `Notifier.build()` is synchronous and reads profile/lives/free-life repos directly. Making those repos return `Future` reintroduces loading states and breaks `build()`. Keep them sync (cache-over-prefs); only genuinely-remote concerns (leaderboard) are async via `FutureProvider`.
- **Keep new gameplay/visuals on the game root**, not `game.world` (see coordinate-space gotcha above) — silent mis-rendering otherwise.
- **Adding a dependency or asset needs a full restart** (`flutter run` again), not hot reload. Hot reload silently won't pick up `pubspec.yaml` changes or new asset registrations.
- **`google_fonts` (Baloo 2 / Nunito, used on Home) fetches font files over the network on first use and caches them.** First launch on a fresh device/emulator needs internet or the text falls back to the platform font for a frame; screenshots taken too early can miss the real face. To ship fully offline, bundle the `.ttf`s as assets and load them via `GoogleFonts.config` / a bundled `TextTheme` instead of runtime fetch.
- **`flutter create` overwrites `lib/main.dart`, `test/widget_test.dart`, `README.md`** with templates — re-read before editing, and expect to replace the generated `MyApp` smoke test.
- **Builder params:** use Dart's wildcard `_` for unused `ValueListenableBuilder`/builder args (`(_, value, _)`), not `__` — the analyzer flags repeated-underscore names.
- **Responsive rows of cards:** don't put a fixed number of equal cards (e.g. Shop coin packs) in a horizontal `ListView` — on wide screens (Pixel 7 Pro) they slide, on small ones a fixed inner height/width overflows (RenderFlex "overflowed by N pixels on the bottom"). Use a `Row` of `Expanded` cards inside `IntrinsicHeight`, with `FittedBox(scaleDown)` around variable-length text (coin amounts, price labels). Fits every width, no scroll, no clip.
- **Continue/revive UI must not look like an error:** the out-of-HP sheet originally used a big red broken-heart (`AppColors.heart`) and "Out of HP!" — players read red-on-panel as a crash/error. Use a positive cue (green `AppColors.neon` + a replay icon + "Keep going?"). Same rule for any non-fatal prompt: green/accent, not red.
- **Never animate a large-sigma `ImageFilter.blur` per frame** — it tanks the frame rate and makes the game unplayable. The animated background orbs (`LiquidBackground` in [theme.dart](lib/app/theme.dart)) once used `ImageFilter.blur(sigma 75)` recomputed every frame; because `LiquidBackground` wraps **every** screen via `MaterialApp.builder`, this ran during gameplay too, and also defeated caching of the HUD's `BackdropFilter` glass (a backdrop blur over an animating background can't be cached). Fix: soft glows use a `RadialGradient` (color → transparent) circle, which is GPU-cheap even when animated; the orb layer is wrapped in a `RepaintBoundary` to isolate its repaints. General rule: prefer `RadialGradient`/`BoxShadow` over `ImageFilter.blur` for glows, keep `BackdropFilter` count low, and verify perf in **profile** mode (`flutter run --profile`), not debug — watch `app_time_stats` for the ~16 ms/60 fps budget.
- **Never run `MaskFilter.blur` per frame in a Flame component's `render()`** — same trap as above, in the engine. Each [bubble.dart](lib/game/components/bubble.dart) painted 3 blurred layers + a `RadialGradient.createShader` allocation *every frame, per bubble* (15–45 gaussian blurs/frame with a full screen), the dominant gameplay-jank source. A component whose pixels depend only on fixed inputs (here `radius`+`color`, never the frame) should be **rasterized once into a `ui.Image`** in `onLoad` (`PictureRecorder` → `picture.toImage(w,h)`, padded for any glow that bleeds past the circle) and blitted with `drawImageRect` each frame; `dispose()` the image in `onRemove`. Keep a cheap no-blur fallback in `render()` for the ~1 frame before the async image is ready. Same rule for particles — [pop_effect.dart](lib/game/components/pop_effect.dart) dropped its per-particle `MaskFilter.blur` (it multiplied across every particle of every overlapping pop during combos) for a plain additive circle / radial-gradient fade.
- **Freeze the animated background during a round.** [GameScreen](lib/presentation/screens/game_screen.dart) bumps a global `activeGameplayCount` ([theme.dart](lib/app/theme.dart)) in `initState`/`dispose`; `LiquidBackground` stops its `AnimationController` while it's > 0. An animating backdrop invalidates every HUD `BackdropFilter`'s cache → ~4 full blur recomputes/frame on top of the game loop. Holding the orbs still (look preserved) lets the glass HUD cache. Mutating this plain `ValueNotifier` in a widget life-cycle is fine — the "never mutate in `initState`" rule is about **Riverpod providers**, not arbitrary notifiers.
- **Small Flame tap targets need an expanded hit zone.** A `CircleComponent`'s tap test equals its visual radius, so the smallest bubbles (radius 22 / 44px) sat at the touch-target floor and the fingertip occluded them — they felt unpoppable. Override `containsLocalPoint` to use `max(radius, _minHitRadius)` (34px ≈ 68px target); visual size unchanged. Flame calls `containsLocalPoint` with no separate AABB cull, so a hit radius larger than the component's `size` works.

### Environment: Windows Application Control can block Dart

Mid-session, `flutter analyze`/`run` started failing with `ProcessStarter::StartForExec failed: An Application Control policy has blocked this file` — a Windows security policy (Smart App Control / WDAC / AV) blocking the Dart runtime from spawning helper processes. This is **environmental, not a code error**. Remedy: allow-list `C:\flutter` (and `…\AppData\Local\Pub\Cache`, the project dir, `…\AppData\Local\Temp`, `C:\Android\Sdk`, `…\.gradle`) in Defender exclusions, or turn off Smart App Control; then restart the terminal. Don't chase it as a build/code bug.

## Windows build gotcha

`android/gradle.properties` sets `kotlin.incremental=false`. This is required, not optional: the project lives on `D:\` while the Flutter Pub cache is on `C:\`, and any Flutter plugin shipping Kotlin sources (e.g. `audioplayers_android`, `shared_preferences_android`) otherwise fails the Android build with `Could not close incremental caches ... different roots` — Kotlin can't compute relative paths across Windows drive roots. If a Gradle build fails with that error, confirm the flag is present, then `flutter clean` and rebuild. iOS builds (macOS only) are unaffected.
