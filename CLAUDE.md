# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Bubble Splash is a Flutter game with a Flame-powered core and a full meta-game around it: a core loop (pop → score → XP → progression), Home/Profile/Leaderboard/Shop/Results screens, and retention hooks (lives-as-continues, watch-ad-for-life, the "Free Life" cooldown reward, achievements, leaderboard). Built with `flame` 1.37 + `flutter_riverpod` 3. **Android is the only active shipping target.** iOS is **parked indefinitely** (maybe years out, maybe never) — the `ios/` scaffold stays committed (costs nothing, ready if it's ever revived) but do **not** invest effort in iOS: no iOS store build, and iOS AdMob is left on Google's **test** placeholder IDs (real iOS ad IDs only matter if an iOS release ever ships). Both platforms are still scaffolded (`android/`, `ios/`); iOS was added via `flutter create --platforms=ios .`; iOS bundle id is `com.bubblesplash.game` (Android applicationId: `com.bubblesplash.game`). iOS **compile/run requires macOS + Xcode**; this repo is developed on Windows, so iOS can be scaffolded/committed here but only built on a Mac. Dart code is platform-agnostic (no `Platform.`/`dart:io`/`MethodChannel` branches). **Windows is NOT a target platform.** A stray `windows/` directory may appear (ephemeral tooling output from `flutter` on this Windows dev box); it holds no `runner/`, is not an enabled platform, and should be ignored, not built against.

**Economy model (important — easy to get wrong, was deliberately changed):** Play is **always free**; lives never gate starting a round. A life is an in-round **continue**: when round HP depletes the player spends a banked life (or watches an ad) to revive. Every life source — passive regen (1 per 30 min, timestamp-based), the home **Free Life** ad claim and in-round revive ads (both owned by `RewardedAdManager`, see the rewarded-ads flow below + `REWARDED_ADS.md`), and Shop **life packs** (`kLifePacks`, bought with coins via `LivesController.addLives`) — banks up to the **single** `LivesState.maxLives` cap (100). A two-cap design (free ceiling 10 + purchase cap 100) was tried and **deliberately reverted** — it confused players; don't reintroduce it. Purchase rules: a life pack must fit *entirely* under the cap or it's refused un-charged (never sell a clamped partial fill), every purchase (coins **and** lives) goes through a confirmation dialog, and ad offers (home Free Life card, continue-sheet ad button) are disabled while the bank is full. `spendLife` from a full bank re-anchors the regen clock (see `lives_controller_test.dart` — dropping below the cap must not instantly refill from a stale timestamp). **Coins are NOT earned by playing** — they're a *purchasable* currency (`PurchaseService`, fake impl now / RevenueCat later) spent on Shop **life packs**. **Coin (real-money) purchases require an account**: a guest tapping a coin pack gets `showSignInPrompt` (in `auth_panel.dart`) at the moment of intent and, on success, continues straight into the purchase confirm; life packs (coin-spend, no real money) stay open to guests. Bubble skins are no longer sold in the Shop (the skin/palette system still exists in code and the game reads the equipped skin; there's just no purchase UI). A round grants XP/progression only.

**Key docs:** `docs/design/candy_cosmos_handoff.md` — the pixel-level UI spec (source of truth for any visual work on Home/Gameplay/Get Ready/Keep Going/Round Over); `docs/CANDY_COSMOS_MIGRATION.md` — the UI-migration log (what's done, what's pending, per-file change history); `PERF_NOTES.md` — the (completed) gameplay-perf/touch pass checklist whose fixes are summarized in the gotchas below.

## Commands

```bash
flutter pub get                 # fetch dependencies
flutter run -d emulator-5554    # run on Android emulator
flutter run --profile -d emulator-5554   # profile build — the ONLY valid way to judge fps; watch `app_time_stats` (~16ms budget). Debug overstates jank.
flutter analyze                 # static analysis (lints from analysis_options.yaml)
flutter test                    # run all tests
flutter test --plain-name "popping a bubble increases the score"   # run a single test by name
dart run tool/gen_audio.dart    # regenerate assets/audio/*.wav sound effects
adb -s emulator-5554 shell pm clear com.bubblesplash.game   # wipe saved state (profile/lives/free-life) to verify a fresh install
adb -s emulator-5554 exec-out screencap -p > shot.png        # headless visual check; `adb shell input tap X Y` drives the UI (screen 1440x3120)
```

`pubspec.yaml` pins `audioplayers_android` to the audioplayers git `main` branch via `dependency_overrides` (added alongside the economy rework; the reason isn't recorded — assume the then-released plugin was broken for this setup). `flutter pub get` therefore needs network access to GitHub, and the override shouldn't be dropped without verifying a full Android build against the released plugin.

After changing pure Dart/Flutter code, hot reload (`r` in the `flutter run` session) is enough. Adding a **dependency or asset** requires a full restart (`flutter run` again), not hot reload. All persistent state lives in `SharedPreferences` (keys `profile`/`lives`/`rewarded_ad`); `pm clear` (above) is the way to reset to a brand-new-player state for manual verification.

## Architecture

Layered, with strict dependency direction `presentation → application → domain ← data`, plus an isolated `game/`. Each layer maps to a folder under `lib/`:

- **`domain/`** — pure Dart, no Flutter/plugin imports. Models (`PlayerProfile`, `LivesState`, `GameResult`/`RewardSummary`, `LeaderboardEntry`, `RewardedAdMeta`, `BubbleSkin`, `Achievement`), repository/service **interfaces** (incl. `PurchaseService` + `CoinPack` catalog, and `RewardedAdProvider` + `RewardedAdLimits`), and catalogs. Achievement unlock conditions and the level math live here as pure functions, so they're trivially unit-testable. Colors are stored as ARGB `int` to keep Flutter out.
- **`data/`** — implements the domain interfaces: `local/` prefs-backed repos (sync), `fake/` simulated-remote leaderboard (seeded, async with latency), `services/` `AdMobRewardedAdProvider` (the **live** `RewardedAdProvider`, wraps `google_mobile_ads`) alongside `FakeRewardedAdProvider` (kept for manual failure-mode testing — simulates AdMob's load-delay / no-fill / fail-to-show / skip-vs-watch, config via `FakeRewardedAdConfig`) + fake purchase (IAP) + no-op notification scheduler + the two `AuthService`s (`SupabaseAuthService` real email/password, `FakeAuthService` in-memory for offline dev/tests).
- **`application/`** — Riverpod controllers own all meta-state: `ProfileController`, `LivesController`, `RewardedAdManager`, `leaderboardProvider`, `GameSessionController`. **`providers.dart` is the single swap point** — the only place concrete `data/` implementations are named. Swapping the mocks for Firebase/RevenueCat means editing only that file. **Ads are live**: `rewardedAdProviderProvider` binds `AdMobRewardedAdProvider` (real `google_mobile_ads`); ad-unit IDs live in `lib/app/ad_config.dart` (test unit in `kDebugMode`, real in release — never click a live ad on your own account), App IDs in `AndroidManifest.xml` + `Info.plist`. iOS IDs are still test placeholders (no iOS AdMob app yet); cap counters are still spoofable prefs — both flagged for pre-launch in `REWARDED_ADS.md`.
- **Auth is real (Supabase email/password).** `AuthService` = `signUp`/`signIn` (return the `AuthAccount`, throw `AuthFailure(message)` on any expected failure) + `signOut`. `authServiceProvider` in `providers.dart` picks `SupabaseAuthService` when `BackendConfig.isConfigured` (Supabase URL + publishable key pasted into `lib/app/backend_config.dart`), else `FakeAuthService` — so the app runs offline. `main()` calls `Supabase.initialize` only when configured. `AuthController.signUp/signIn` return `String?` (null = success, else the error message to show). Sign up collects a display **name** + **country** (ISO code, prefilled from device locale, stored in Supabase user metadata → `AuthAccount.country`, for the future local leaderboard). Supabase's **"Confirm email"** setting must be **off** for straight-through signup, else sign-up returns no session and the UI says "check your email to confirm". No Google / OAuth — deliberately dropped in favour of email/password (works on Android+iOS with zero Google Cloud setup).
- **Supabase DB schema** lives in `supabase/migrations/` (applied to the linked project via `npx supabase db push`; `supabase/config.toml` is the CLI link). Two tables, both RLS "owner-only" (a player reads/writes only their own row via `auth.uid()`): **`profiles`** — one row per account, auto-created on sign-up by the `handle_new_user` trigger (seeds `name`+`country` from the auth metadata `signUp` sends). Columns mirror the domain models exactly: identity (`name`, `country`, `avatar_emoji` [icon key], `avatar_color` [ARGB bigint]), `coins`, progression (`xp`, `high_score`, `games_played`, `total_bubbles_popped`, `best_streak`), `unlocked_achievement_ids text[]`, lives bank (`lives_count`, `lives_last_regen_ms`), free-life cooldown (`free_life_last_claim_ms`), and the **server-authoritative rewarded-ad counters** (`ad_daily_count`, `ad_daily_window_start_ms` — added in `0003_ad_limits.sql`; the home cooldown reuses `free_life_last_claim_ms`), enforced by the `ad_limit_state()` / `claim_ad_view(kind)` RPCs (see the rewarded-ads anti-spoof note below). **No skin columns** — skins aren't sold, so equipped/owned stay local `'classic'` defaults, never persisted server-side. **`game_rounds`** — append-only per-round log mirroring `GameResult` (`score`, `bubbles_popped`, `max_combo`, `golden_popped`), immutable (insert+select only). The **`record_round(score, bubbles_popped, max_combo, golden_popped)`** RPC logs a round *and* rolls it into the profile aggregates in one authenticated call. The public **`leaderboard`** view is a narrow read-only projection (name/country/avatar/`high_score`/`total_bubbles_popped`) granted to anon+authenticated — never exposes coins/lives/email.
- **Profile ↔ DB sync** (`RemoteSyncService`, `application/providers.dart` → `remoteSyncServiceProvider`): local prefs stay the **source of truth**; Supabase is a **best-effort mirror** for signed-in accounts only. `SupabaseRemoteSyncService` when `BackendConfig.isConfigured`, else `NoopRemoteSyncService` (guests / offline / tests never touch the network). Flow: **sign in** pulls the `profiles` row into local storage before `ProfileController` rebuilds (so records follow the account across devices); **sign up** and every profile `_commit` upsert the row (coins/xp/stats/name/avatar/achievements); the fresh tagged-name profile is pushed from `ProfileController.build` (not from `AuthController`, which would create an auth↔profile provider **circular dependency**); **round end** appends a `game_rounds` row via `logRound`. All remote calls fail soft — reads return null, writes are fire-and-forget and swallow errors, so gameplay never blocks on the network. **Lives/free-life are NOT synced** — they stay device-global retention mechanics (their DB columns exist but the app leaves them at defaults). `equippedSkinId`/`ownedSkinIds` are never round-tripped (local `'classic'` defaults).
- **`presentation/`** — `ConsumerWidget` screens (`home`, `game`, `profile`, `leaderboard`, `shop`) and widgets. No widget reads a repository directly; everything goes through controllers. **The whole game flow — Home, Gameplay HUD, Get Ready, Keep Going sheet, Round Over — is skinned to an external design handoff ("Candy Cosmos")**: spec is in the repo at `docs/design/candy_cosmos_handoff.md`, migration log at `docs/CANDY_COSMOS_MIGRATION.md`. Shared tokens/widgets live in **`lib/app/candy.dart`** (`Candy` colors + `display`/`ui` text styles [Baloo 2 / Nunito via `google_fonts`], `CandyNebulaBackground`, `CandyGlass`, `CandyChip`, `CandyCtaButton`, `CandySheet`) — use these, don't re-declare hexes. Every spec px value is from the handoff's **322px-wide reference frame** and multiplied by `candyScale(context) = screenWidth / 322` — keep new dimensions in that same `* s` space, don't hardcode logical px. Candy surfaces are plain translucent fills (deliberately **no `BackdropFilter`** — cheap, cacheable). **Every screen (home, game flow, profile + dialogs, shop, leaderboard) is now Candy Cosmos** — the legacy glass widgets (`glass.dart`, `status_badges.dart`) and `theme.dart` (AppColors, `LiquidBackground` orb backdrop) were deleted; the minimal Material theme (Candy-seeded, snackbar style) lives in `app.dart`. Shared candy widgets also include `CandySectionLabel`, `CandyStatPill`, `CandyBackCircle`, `CandyBubbleCluster`/`CandyFloatBubble` (the animated bubble-cluster logo, shared by Home and Login), `CandyGameTitle` (the glowing BUBBLE/SPLASH title), and `candyBubbleGradient`/`candyBubbleShades` (the glossy avatar recipe, used by `PlayerAvatar` and the profile picker). Email/password auth UI lives in `presentation/widgets/auth_panel.dart`: `AuthPanel` (the Sign in / Sign up form — mode toggle, name/email/password fields, country selector on sign-up, inline validation + error line, orange CTA) and `showSignInPrompt` (the Candy-sheet "sign in to continue" gate that embeds `AuthPanel`, used by the Login screen, the Shop coin gate, and the guest name lock). The country list + searchable picker (`showCountryPicker`) live in `lib/app/countries.dart`.
- **`game/`** — the Flame engine, kept **Riverpod-free**. `GameScreen` injects `onGameOver(GameResult)`, `onContinueOffer()`, and the equipped skin's palette. The game reports outcomes up; the screen bridges to controllers and shows the continue sheet. In-round state (`score`, `hp`, `combo`, `soundOn`) stays as `ValueNotifier`s consumed by `GameHud`/`ResultsOverlay`.

Key flows:
- **Sync vs async repositories:** profile/lives/rewarded-ad repos are **synchronous** (prefs is warmed in `main()` and injected via `sharedPreferencesProvider.overrideWithValue`), so meta screens never flash a loading state and Riverpod `Notifier.build()` can read them directly. Only the leaderboard repo is async (it models the network) and is consumed via `FutureProvider` with `.when(...)`.
- **Leaderboard is two-metric:** boards rank by a `LeaderboardMetric` — `highScore` (peak single-round score) or `totalPops` (lifetime bubbles popped) — crossed with `LeaderboardScope` (local/global). `leaderboardProvider` is a `FutureProvider.autoDispose.family` keyed on the record `(scope, metric)`; the repo only supplies bots per scope, and the **controller** merges the player, sorts by `entry.valueFor(metric)`, and assigns ranks (sorting lives in the controller, not the repo). Fake bots seed `bubblesPopped` loosely correlated to `score` with heavy jitter so the two boards rank people differently. Add a new ranking metric by extending the enum + `valueFor`, not by adding a repo method.
- **Continue flow:** depleting round HP (or popping a bomb) calls `_offerContinue()` → `pauseEngine()` (guarded by `isMounted` so headless tests don't touch a missing game loop) + `onContinueOffer()`. The screen shows `continue_round_sheet.dart`; the player continues with a banked life or watches a rewarded ad through `RewardedAdManager` (each completed view banks +1 life), then `game.continueRound()` restores HP, **clears all on-screen bubbles**, and grants a `headStartSeconds` (3s) breather — bubbles resume at **50% speed** (`reliefFactor`, the player just died at full speed) and recover linearly to full over `reliefRecoverySeconds` (45s); score/progression are untouched. The `headStart` `ValueNotifier` drives the full-screen "GET READY 3·2·1" overlay (`_HeadStartOverlay` in `game_screen.dart`). Declining calls `game.finishRound()` (emits the `GameResult`). `_awaitingDecision` freezes taps/spawns while the sheet is up.
- **Difficulty plateaus — deliberately (retention):** spawn speed/interval ramp with `1 - exp(-score/k)` curves (`rampSpeedBonus`, `spawnIntervalFor` — pure statics, unit-tested) that match the old linear ramp's early slope but cap at +240 px/s and a 0.38s spawn floor; at most `_maxOnScreen` (6) bubbles exist at once (the spawner waits, so density stays humanly clearable); and a continue restarts at 50% speed (`reliefFactor`) recovering to full over 45s (`recoverRelief`). Don't reintroduce unbounded ramps — `test/difficulty_test.dart` pins the caps. Taps also lag-compensate: a bubble's hit zone stretches *downward* by ~100ms of its travel (`Bubble.containsLocalPoint`) because fingers aim at where a fast bubble was.
- **Lives regen is timestamp-based** (`LivesState.lastRegenAtMs`): `LivesController` recomputes earned lives from elapsed time on load and on a 1-second timer, so regen is correct while the app is closed. A separate `livesTickerProvider` drives per-second countdown UIs without churning controller state. `clockProvider` injects the clock so time logic is deterministically testable.
- **Rewarded ads — `RewardedAdManager` owns all policy** (full spec: `REWARDED_ADS.md`). It talks to ads only through the `RewardedAdProvider` interface; the live impl is `AdMobRewardedAdProvider` (`google_mobile_ads`), bound at `rewardedAdProviderProvider` (swap back to `FakeRewardedAdProvider` there to test failure modes offline). Limits live in one file (`RewardedAdLimits`): **3 revives per death event**, **30-min** home cooldown, **20 completed views/day** (rolling 24h). Persisted state (`RewardedAdMeta`, prefs key `rewarded_ad`) carries the daily-count window + home cooldown, with clock-tamper guards (future/backward clock ⇒ cooldown/cap stay **active**, never reset). **Prefs is spoofable, so the daily cap + home cooldown are now enforced server-side for signed-in users** (anti-spoof "Piece 1"): the manager hydrates its counters from Supabase on build and authorizes each completed view through the `RewardedAdGate` (`rewardedAdGateProvider` → `SupabaseRewardedAdGate` / `NoopRewardedAdGate`), which calls the `ad_limit_state()` / `claim_ad_view(kind)` RPCs (migration `0003_ad_limits.sql`, server-clock authoritative). The gate **fails soft** — guests, offline, and a backend-down state fall back to the local `RewardedAdMeta` path unchanged, and a life is granted only on the server's `granted:true`. Full lock-down (anonymous-auth for guests + AdMob SSV) is deferred — reward is lives, no real money. See `REWARDED_ADS.md` → "Anti-spoof". Button UI is a strict state machine (`READY/LOADING/COOLDOWN/NO_FILL/CAP_REACHED/CONSUMED`), driven only by manager state, never a local bool — no dead buttons. **Reward is granted at exactly one choke point (`_grantReward`) and only on `RewardedAdShowResult.rewardEarned`** — never on tap/dismiss; idempotent (provider is single-use). Two intended behaviours worth noting: the **3-ad cap resets every death** (`beginDeathEvent`), so reviving then dying again gives a fresh 3 (the 20/day is the global backstop); and **skipping an ad does NOT burn a revive slot or the daily count** — only a completed watch moves any counter (matches real AdMob: no reward, no charge, no cost). Ads are **preloaded proactively** (run start, low health, prompt open) with NO_FILL exponential backoff (1→2→4→8…60s), never at the moment of death. Engine pause/resume is owned by the continue flow (`_offerContinue`→pause, both `continueRound`/`finishRound`→resume once); `game_screen` re-pauses on foreground while `game.isAwaitingDecision` so backgrounding mid-ad can't leave the loop running or double-resume.
- **Bubble kinds:** `Bubble` carries a `BubbleKind` (normal/golden/bomb). Golden = bonus points; popping a bomb depletes the round (continue offered); letting a bomb escape is harmless. Combo multiplier rises with consecutive pops within `comboWindow`; each scoring pop spawns a floating `ScorePopup` ("xN") on the game root, and the HUD's live combo pill ("COMBO 3× ·12") shows multiplier + chain while `combo >= 2`. Bubbles render with the Candy gloss recipe (white specular → light → mid → dark, light/dark derived from the palette color in `_paintCandy`); bombs are a translucent glass ball with a **drawn** cartoon bomb (`_paintBomb`) — never an emoji glyph (tofu on old Androids).
- **Avatars are Material icons, not emoji** (`player_avatar.dart`, `kAvatarIcons`/`avatarIconFor`). Old Androids often lack a color-emoji font and render blank tofu boxes, so the profile `avatarEmoji` field stores an icon *key* string. Don't reintroduce raw emoji glyphs in rendered UI for the same reason.
- **Player names are unique** (`ProfileController`): a stable 4-digit `#tag` derived from the profile id is appended (Discord-style), so duplicate base names never collide. Fresh base names are readable — `Guest` for guests, the account's chosen display *first name* when signed in (e.g. `Guest#0421`, `Ammar#7305`) — never a generic `Player`. The tag is re-applied on every `rename`. **Guests cannot rename**: custom names are an account perk — `rename` no-ops unless signed in (`canRename` guard), and the Profile screen routes a guest's name tap to `showSignInPrompt` instead of the name editor (opening the editor right after a successful sign-in). Avatar icon/color stays editable for guests.

- **Coordinate space gotcha:** bubbles and pop effects are added directly to the game root (`game.add(...)`), **not** to `game.world`. They're positioned in screen coordinates using `size`. Adding a visual to `game.world` would put it in camera/world space and render in the wrong place — keep new gameplay visuals on the game root.

### Headless-test safety

Tests drive controllers and the Flame game directly without a widget tree, so anything needing platform plugins must be guarded:

- Controller/repository tests use a `ProviderContainer` with `sharedPreferencesProvider.overrideWithValue(...)` (`SharedPreferences.setMockInitialValues({})`) and `clockProvider.overrideWithValue(() => now)` for deterministic time. See `test/lives_controller_test.dart` for the pattern.
- The game's audio (`FlameAudio`) is guarded by `isMounted` + `try/catch` so it no-ops headlessly. **Preserve these guards** when adding sound, or `test/widget_test.dart` will crash on missing plugins.

### Audio assets

Sound effects are generated, not committed by hand: [tool/gen_audio.dart](tool/gen_audio.dart) synthesizes `assets/audio/pop.wav` and `game_over.wav` from pure math using only the Dart SDK. Edit the synthesis functions there and rerun `dart run tool/gen_audio.dart` to change the sounds. `assets/audio/` is registered in `pubspec.yaml`; `FlameAudio`'s default prefix is `assets/audio/`.

## Gotchas & lessons learned

Hard-won rules from building this — follow them when adding features to avoid repeating the same bugs:

- **Never mutate a Riverpod provider inside a widget life-cycle** (`initState`, `build`, `didChangeDependencies`, `dispose`). Doing so throws *"Tried to modify a provider while the widget tree was building."* This bit us when `GameScreen.initState` called `livesController.spendLife()` (which sets state). Fix pattern used in [game_screen.dart](lib/presentation/screens/game_screen.dart): defer the mutation with `WidgetsBinding.instance.addPostFrameCallback((_) { if (mounted) ... })`, or do it in a callback (e.g. a button's `onPressed`). **Reading** a provider in `build` (`ref.watch`/`ref.read`) is fine — only *mutation* is forbidden.
- **Don't make the sync repos async to "be safe."** `Notifier.build()` is synchronous and reads profile/lives/free-life repos directly. Making those repos return `Future` reintroduces loading states and breaks `build()`. Keep them sync (cache-over-prefs); only genuinely-remote concerns (leaderboard) are async via `FutureProvider`.
- **Keep new gameplay/visuals on the game root**, not `game.world` (see coordinate-space gotcha above) — silent mis-rendering otherwise.
- **Adding a dependency or asset needs a full restart** (`flutter run` again), not hot reload. Hot reload silently won't pick up `pubspec.yaml` changes or new asset registrations.
- **`google_fonts` (Baloo 2 / Nunito, used on Home) fetches font files over the network on first use and caches them.** First launch on a fresh device/emulator needs internet or the text falls back to the platform font for a frame; screenshots taken too early can miss the real face. To ship fully offline, bundle the `.ttf`s as assets and load them via `GoogleFonts.config` / a bundled `TextTheme` instead of runtime fetch.
- **`flutter create` overwrites `lib/main.dart`, `test/widget_test.dart`, `README.md`** with templates — re-read before editing, and expect to replace the generated `MyApp` smoke test.
- **Builder params:** use Dart's wildcard `_` for unused `ValueListenableBuilder`/builder args (`(_, value, _)`), not `__` — the analyzer flags repeated-underscore names.
- **Responsive rows of cards:** don't put a fixed number of equal cards (e.g. Shop coin packs) in a horizontal `ListView` — on wide screens (Pixel 7 Pro) they slide, on small ones a fixed inner height/width overflows (RenderFlex "overflowed by N pixels on the bottom"). Use a `Row` of `Expanded` cards inside `IntrinsicHeight`, with `FittedBox(scaleDown)` around variable-length text (coin amounts, price labels). Fits every width, no scroll, no clip.
- **Continue/revive UI must not look like an error:** the out-of-HP sheet originally used a big red broken-heart and "Out of HP!" — players read red-on-panel as a crash/error. Use a positive cue (today: the warm orange `Candy.orangeChip` restart circle + "Keep going?"). Same rule for any non-fatal prompt: warm/accent colors, not red.
- **Never animate a large-sigma `ImageFilter.blur` per frame** — it tanks the frame rate and makes the game unplayable. The (since-deleted) `LiquidBackground` orb backdrop once used `ImageFilter.blur(sigma 75)` recomputed every frame; because it wrapped **every** screen via `MaterialApp.builder`, this ran during gameplay too, and also defeated caching of the HUD's `BackdropFilter` glass (a backdrop blur over an animating background can't be cached). Fix then: `RadialGradient` (color → transparent) glows in a `RepaintBoundary`. Fix now: the layer is gone entirely — every screen paints its own static `CandyNebulaBackground`, and an app-wide animated backdrop under opaque screens is pure wasted work; don't reintroduce one. General rule: prefer `RadialGradient`/`BoxShadow` over `ImageFilter.blur` for glows, keep `BackdropFilter` count low, and verify perf in **profile** mode (`flutter run --profile`), not debug — watch `app_time_stats` for the ~16 ms/60 fps budget.
- **Never run `MaskFilter.blur` per frame in a Flame component's `render()`** — same trap as above, in the engine. Each [bubble.dart](lib/game/components/bubble.dart) painted 3 blurred layers + a `RadialGradient.createShader` allocation *every frame, per bubble* (15–45 gaussian blurs/frame with a full screen), the dominant gameplay-jank source. A component whose pixels depend only on fixed inputs (here `radius`+`color`, never the frame) should be **rasterized once into a `ui.Image`** in `onLoad` (`PictureRecorder` → `picture.toImage(w,h)`, padded for any glow that bleeds past the circle) and blitted with `drawImageRect` each frame; `dispose()` the image in `onRemove`. Keep a cheap no-blur fallback in `render()` for the ~1 frame before the async image is ready. Same rule for particles — [pop_effect.dart](lib/game/components/pop_effect.dart) dropped its per-particle `MaskFilter.blur` (it multiplied across every particle of every overlapping pop during combos) for a plain additive circle / radial-gradient fade.
- **Mutating a plain `ValueNotifier` in a widget life-cycle is fine** — the "never mutate in `initState`" rule above is about **Riverpod providers**, not arbitrary notifiers. (The old example, `activeGameplayCount` freezing `LiquidBackground` during rounds, was deleted along with that background layer.)
- **Small Flame tap targets need an expanded hit zone.** A `CircleComponent`'s tap test equals its visual radius, so the smallest bubbles (radius 22 / 44px) sat at the touch-target floor and the fingertip occluded them — they felt unpoppable. Override `containsLocalPoint` to use `max(radius, _minHitRadius)` (34px ≈ 68px target); visual size unchanged. Flame calls `containsLocalPoint` with no separate AABB cull, so a hit radius larger than the component's `size` works.

### Environment: Windows Application Control can block Dart

Mid-session, `flutter analyze`/`run` started failing with `ProcessStarter::StartForExec failed: An Application Control policy has blocked this file` — a Windows security policy (Smart App Control / WDAC / AV) blocking the Dart runtime from spawning helper processes. This is **environmental, not a code error**. ## Windows build gotcha

`android/gradle.properties` sets `kotlin.incremental=false`. This is required, not optional: the project lives on `D:\` while the Flutter Pub cache is on `C:\`, and any Flutter plugin shipping Kotlin sources (e.g. `audioplayers_android`, `shared_preferences_android`) otherwise fails the Android build with `Could not close incremental caches ... different roots` — Kotlin can't compute relative paths across Windows drive roots. If a Gradle build fails with that error, confirm the flag is present, then `flutter clean` and rebuild. iOS builds (macOS only) are unaffected.
